1 Cr&#230;ft
2 ==========
3 
4 This is a compiler for a new systems programming language, written for Caltech
5 CS 81 with Donnie Pinkston as project mentor. The language, Cr&#230;ft, is a
6 strongly statically typed, imperative, low-level programming language with
7 support for generic programming inspired by C and Rust.
8 
9 Internal documentation for the compiler can be found
10 [here](http://iankuehne.com/craeft).
11 
12 Language
13 ========
14 
15 Cr&#230;ft is a systems programming language based on:
16 - Low-level access to the hardware. A competent programmer should understand
17  how their code is executed on the processor. In practice, this means that
18  memory is manually managed, and the programmer can make unlimited use of
19  pointers.
20 - Low-cost abstractions. The programmer should not have to balance performance
21  with maintainable code.
22 - Strong static typing. The type system should be expressive, and the compiler
23  should only allow it to be circumvented with explicit annotations.
24 
25 Syntax
26 ------
27 
28 The syntax is simple, and largely familiar to any C programmer. Roughly, in
29 EBNF:
30 
31 ```
32 identifier: [a-z|non-ascii][a-zA-Z0-9_|non-ascii]*
33 
34 TypeName: [A-Z][a-zA-Z0-9_|non-ascii]*
35 
36 Type: TypeName
37  | TypeName<:[Type,]* Type:>
38  | Type *
39 
40 op: [!*+-><&%^@~/=]+
41 
42 expr: identifier
43  | literal
44  | expr op expr
45  | identifier ( [expr,]* expr)
46  | identifier ( )
47  | Type ( expr )
48 
49 declaration: Type identifier;
50  | Type identifier = expr;
51 
52 statement: declaration
53  | expr;
54  | ifblock
55  | return expr;
56 
57 ifblock: if expr { statement* }
58  | if expr { statement* } else { statement* }
59 
60 arglist: ( )
61  | ([Type identifier,]* Type identifier)
62 
63 typelist: <: :>
64  | <: [Type,]* Type :>
65 
66 signature: fn identifier arglist -> Type
67  | fn typelist identifier arglist -> Type
68 
69 func: signature;
70  | signature { statement* }
71 
72 struct: struct Type { [Type identifier;]* }
73  | struct typelist Type { [Type identifier;]* }
74 
75 toplevel: func | struct
76 
77 program: toplevel*
78 ```
79 
80 This omits a number of details important to the semantics, in particular which
81 operators are actually allowed (all C operators except the ternary operator and
82 the assignment operators) and the precedences and fixities (which follow C).
83 I've also omitted the definition of `literal`, simply because it is boring and
84 new literals are likely to be added soon. C numeric literals are currently
85 supported, except for hexadecimal literals. Double-quoted strings are also
86 supported.

88 An Example
89 ----------
90 
91 As is traditional, we introduce the language with a definition of the factorial
92 function:
93 
94 ```
95 fn fact(U64 n) -> U64 {
96  if n == 0 {
97  return 1;
98  }
99  return n * fact(n - 1);
100 }
101 ```
102 
103 A couple of interesting features of the language are present here: integer types
104 are written with a `U` or an `I` followed by the number of bits, which can range
105 from 1 (for a boolean) to 64.
106 
107 Also, the compiler optimizes simple tail calls. The code it produces for this
108 function on x86 is (after cleaning up a couple of labels and removing some
109 assembler directives):
110 
111 ```assembly_x86
112 fact:
113  movl $1, %eax
114  jmp finish
115 
116 loop:
117  imulq %rdi, %rax
118  decq %rdi
119 
120 finish:
121  testq %rdi, %rdi
122  jnz loop
123  retq
124 ```

126 Generics
127 --------
128 
129 The main feature of Cr&#230;ft not shared with C is its support for structural
130 and functional generics. A structural template allows filling struct fields
131 with type parameters. So, for example, a generic linked list would look like:
132 
133 ```
134 struct<:T:> ListNode {
135  T contents;
136  U8 *next;
137 }
138 ```
139 
140 (there is a bug in the compiler implementation which prevents recursive struct
141 definitions, even where the size can be computed--this is an early alpha
142 compiler). This can then be used like a normal struct by providing the type
143 parameter in a declaration:
144 
145 ```
146 ListNode<:U64:> int_list;
147 int_list.contents = 10;
148 ...
149 ```
150 
151 Functional generics similarly allow for a function with a type parameter. So,
152 if we wanted a function which created a new singleton list of a generic type, we
153 could write:
154 
155 ```
156 fn<:T:> list_new(T contents) -> ListNode<:T:> {
157  ListNode<:T:> result;
158  result.contents = contents;
159  result.next = (U8 *)0;
160  return result;
161 }
162 ```
163 
164 The compiler currently expands all templates at compile time, so this is
165 nominally a "zero-cost" abstraction. Of course, that could easily lead to an
166 explosion in compile times and code size, so a short term goal is to add support
167 for runtime polymorphism, which has no such drawbacks.
168 
169 Unicode
170 -------
171 
172 Cr&#230;ft supports non-ASCII identifiers and strings by treating any characters
173 which are not valid ASCII as unknown lower-case characters. Thus, code may use
174 non-ASCII UTF-8 characters, or characters from an extended ASCII encoding.
175 
176 Linking with C Code
177 -------------------
178 
179 Linking with C code is trivial. The compiler uses the system C calling
180 convention and alignment rules, so compiled code need simply declare the C
181 function, and then it may be called freely.

197 Usage and Example Code
198 ======================
199 
200 The compiler can produce LLVM IR, assembly, or static object code files, with
201 the `--ll`, `--asm`, and `-c` flags respectively. So to compile the
202 `factorial.cr` example:
203 
204 ```
205 ./craeftc ../examples/factorial.cr -c factorial.o
206 ```
207 
208 Most of the examples have a corresponding C "harness" which calls the Cr&#230;ft
209 functions. So, to compile and run `factorial.cr` with the harness:
210 
211 ```
212 cc ../examples/factorial_harness.c -c
213 cc factorial_harness.o factorial.o -o factorial
214 ./factorial
215 ```
